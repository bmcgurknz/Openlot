# v1.3 enhancements

Five additions on top of the shipped v1.2 (Procore OpenLot, dual
server/static edition, ATS 1120 aligned): lot register provenance and
sorting, claim provenance, a real lot change-history audit trail, a guided
bulk-import wizard for Procore/spreadsheet exports, and this documentation.
This doc is the implementation record; see [architecture.md](architecture.md)
for the system overview these enhancements sit inside.

## 1. Schema changes

`migrations/003_history_and_provenance.sql`, applied after `001_init.sql`
and `002_ats1120.sql`:

| Table | Change |
|---|---|
| `lots` | `+ created_at timestamptz NOT NULL DEFAULT now()` (existing rows backfilled to the migration run time â€” there is no way to recover a true historical creation date for pre-v1.3 rows); `+ builder text`, `+ stage text`, `+ owner text` (first-class subdivision/civil fields â€” a lot's builder/purchaser, delivery stage, and owner, none of which existed before v1.3) |
| `claim_periods` | `+ created_by text` (who created the progress-claim period) |
| `lot_history` (new) | `id uuid PK, project_id bigint, lot_id text, at timestamptz, "user" text, field text, previous_value text?, new_value text?`, indexed on `(project_id, lot_id, at DESC)` |

`src/types.ts` gained `Lot.builder/stage/owner/createdAt`,
`ClaimPeriod.createdBy`, and the `LotHistoryEntry` interface. The
`Repository` interface (`src/db/repository.ts`) gained
`appendHistory(entry)` / `listHistory(projectId, lotId)`, implemented by
both `MemoryRepository` (in-memory array, sorted newest-first on read) and
`PgRepository` (plain insert + `ORDER BY at DESC` select â€” see
`src/db/pg-repository.ts`).

Run `npm run migrate` (or let `src/index.ts` apply it automatically on
server-edition boot) before running `npm run seed` against a real
Postgres instance. Demo mode and tests use `MemoryRepository` and need no
migration.

## 2. Audit implementation (Lot Change History)

**Design decision:** rather than diffing whole-lot snapshots, every
service method that changes a tracked field writes one explicit
`LotHistoryEntry` at the point of change, with the actual previous/new
values it already has in hand. This is simpler, cheaper, and more
precise than snapshot-diffing, at the cost of needing a write at every
call site that mutates a tracked field.

What is tracked, and where the write happens:

| Event | Written by | `field` value |
|---|---|---|
| Status transition | `LotService.transition()` | `"Status"` |
| Superseding a lot | `LotService.transition()` | `"Superseded by"` |
| Hold/witness point released or reinstated | `LotService.releaseHoldPoint()` | `"Hold point"` |
| Bounds (description) redefined | `LotService.update()` | `"Description"` |
| Builder / Stage / Owner changed | `LotService.update()` | `"Builder"` / `"Stage"` / `"Owner"` |
| Notes edited | `LotService.update()` | `"Notes"` |
| Lot added to a progress-claim period | `ClaimService.addLot()` | `"Progress claim"` |
| Progress-claim period issued | `ClaimService.issuePeriod()` | `"Progress claim"` (one entry per claimed lot) |

Every write site accepts an optional `actor` string (there is no login
system in this app â€” actor is a free-text name/role the caller supplies,
matching the existing pattern used for ATS 1120 cl 11.6 hold-point
release). Missing actor falls back to `"unspecified"` rather than
throwing, except where an actor was already mandatory before v1.3 (hold
point release still refuses without one â€” cl 11.6 unchanged).

**Backwards compatibility:** v1.2 already had a cruder, ad-hoc audit
mechanism â€” free-text notes appended to `Lot.notes` for hold-point
release/reinstatement and bounds redefinition (`tests/unit/ats1120.test.ts`
asserts on this). v1.3 does **not** remove that; it adds the structured
`lot_history` write alongside it. Removing the notes-string behaviour
would have been a breaking change for anyone parsing `notes` today, and
the two mechanisms serve different purposes: `notes` is a human-readable
running log a user might also add to directly, `lot_history` is a
queryable, per-field structured trail.

**Surfacing:** `GET /api/projects/:projectId/lots/:lotId` now returns a
`history` array alongside the existing dossier fields, and there's a
dedicated `GET .../lots/:lotId/history` endpoint. The web UI
(`web/src/pages/LotDetailPage.tsx`) adds an Overview/History tab toggle;
the History tab renders the trail newest-first in a plain table (When /
Field / From / To / By) â€” no client-side sorting needed since the
repository already returns it in that order.

## 3. Sorting implementation (lot register)

`web/src/pages/RegisterPage.tsx` defines a `COLUMNS` array â€” one entry
per visible column (Lot ID, Description, Work type, Spec/ITP ref, Cost
code, Builder, Stage, Owner, Qty, Date created, Status) â€” each with an
`accessor(lot) => string | number | null` and a `render(lot) => JSX`.
Clicking a header cycles that column through **default â†’ ascending â†’
descending â†’ default** (`toggleSort`); only one column sorts at a time,
matching the "asc/desc/default toggle" requirement. `null` values always
sort last regardless of direction (a lot with no cost code shouldn't
jump to the top when sorting cost code descending).

This is entirely client-side (`Array.prototype.sort` over the array
`api.lots(projectId)` already returned) â€” no backend change, no new
endpoint, no pagination concerns, because civil-project lot registers run
to the low thousands of rows at most (see Â§7, Scalability notes). Builder
/ Stage / Owner sort exactly like any other column despite being newer,
optional, subdivision-specific fields â€” nothing in the sort mechanism is
Lot-schema-specific beyond the `COLUMNS` accessor list, so any future field
added to `Lot` is sortable by adding one entry.

## 4. Bulk import architecture

Requirement: a guided wizard â€” upload â†’ detect format â†’ validate â†’
preview â†’ resolve issues â†’ confirm â†’ summary â€” supporting CSV and XLSX,
with automatic + manual field mapping, duplicate detection, and
create-or-update semantics.

### Module layout

```
src/lib/import/            â€” pure, edition-agnostic domain logic (Node + browser)
  types.ts                 â€” ImportField, FieldMapping, ImportRowPlan, etc.
  csv.ts                   â€” RFC4180-ish CSV parser (also used by the pre-existing
                              static-edition register importer, web/src/local.ts)
  xlsx.ts                  â€” first-sheet XLSX/XLS â†’ {headers, rows} via SheetJS ("xlsx" package)
  detect.ts                â€” filename-extension detection, zip-magic-bytes fallback
  fields.ts                â€” autoMapColumns(): header text â†’ canonical field, by alias table
  plan.ts                  â€” planImport(): validation + duplicate detection + create/update/skip decision
  run.ts                   â€” runImportPlan() against a LotService directly (used by tests;
                              see web/src/importRunner.ts for what the UI actually calls)

web/src/pages/ImportWizardPage.tsx  â€” the wizard UI (steps below)
web/src/importRunner.ts             â€” runs a plan through the shared `api` client
                                       (works unmodified in both editions â€” see Â§4.4)
```

### 4.1 Format detection

`detectFormat(filename, headBytes?)`: file extension first (`.csv` /
`.txt` / `.tsv` â†’ csv; `.xlsx` / `.xlsm` / `.xls` â†’ xlsx); if the
extension is missing or unrecognised, falls back to the leading bytes â€”
XLSX/XLSM files are zip archives (`PK\x03\x04` magic), anything else is
assumed to be delimited text.

### 4.2 Field mapping â€” automatic + manual

`autoMapColumns(headers)` normalises each header (lowercase, collapse
separators/whitespace) and matches it against an alias table covering the
spellings seen in Procore lot-register exports and common spreadsheets
(e.g. "Pay Item" / "Payment Item" / "Schedule Item" â†’ `paymentItemNumber`;
"Qty" / "Quantity" â†’ `quantity`). Each canonical field matches at most one
column. The wizard's mapping step then lets the user override any guess
via a per-column `<select>` â€” nothing is applied until the user proceeds
to preview, so overrides are cheap to try.

**Canonical import fields** (`IMPORT_FIELDS` in `types.ts`): `lotId`
(required), `description` (required, falls back to the Lot ID if blank â€”
with a warning, not an error), `workType`, `specReference`, `costCode`,
`paymentItemNumber`, `quantity`, `uom`, `status`, `builder`, `stage`,
`owner`, `notes`. `workType` can also be inferred from the Lot ID's
`LOT-[WT]-NNNN` prefix when no explicit column is mapped; if both are
present and disagree, the ID prefix wins (with a warning).

### 4.3 Validation, duplicate detection, create-or-update

`planImport(rows, mapping, existingLotIds)` builds one `ImportRowPlan` per
source row, with no side effects (safe to re-run on every mapping/preview
change):

- **Lot ID validation:** must match `LOT-[A-Z]{2}-\d{1,5}`, work type
  must be a known code, and the ID is normalised to the canonical
  4-digit-padded form (`LOT-EW-2` â†’ `LOT-EW-0002`) so an unpadded import
  ID still matches an existing padded one.
- **Duplicate detection** has two layers: (a) within the file â€” the same
  Lot ID appearing twice is an error, second occurrence skipped; (b)
  against the register â€” `existingLotIdSet(existingLots)` is compared
  against each row's normalised Lot ID.
- **Create-or-update action:** a Lot ID not already in the register â†’
  `create`; one that is â†’ `update`; a row with any blocking validation
  error â†’ `skip` regardless of what it would otherwise have been.
- Quantity is parsed leniently (strips thousands separators); a
  non-numeric value is a warning, not an error, and the field is left
  blank rather than blocking the row.

Issues are `error` (blocks the row; checkbox forced off, can't be
re-included) or `warning` (row still runs by default; user can exclude it
in the preview/resolve-issues step, which is the same screen â€” see the
wizard steps below).

**Executing the plan** never clobbers data blindly: for updates, only
fields the source row actually had a value for are sent (`orUndef()` in
`run.ts` / the `maybe()` helper in `web/src/importRunner.ts`) â€” a blank
cell means "no information", not "clear this field". `updateExisting` is
a wizard-level toggle; turning it off makes every `update`-classified row
a no-op (counted as skipped) so the wizard can be used purely to create
new lots from an export that also happens to repeat ones already in the
register.

For a `create` row with a `status` of "work complete" / "work_complete",
the wizard also transitions the new lot to `work_complete` â€” but never to
`conformed` or `closed`, on the same principle the original CSV importer
already used: conformance must be earned through the evidence rules, not
asserted by a spreadsheet.

### 4.4 Wizard steps and edition-agnostic execution

`ImportWizardPage.tsx` steps: **Upload** (file picker, auto-detect on
selection) â†’ **Map fields** (auto-mapped, editable) â†’ **Preview, resolve
issues & confirm** (one screen: counts, per-row action/issues table with
an include checkbox, the "update existing" toggle, and a "recorded as"
actor field) â†’ **Summary** (created/updated/skipped/failed counts, with
per-row error detail for anything that failed at execution time, e.g. a
sequence collision from a concurrent edit).

The wizard is reachable from both editions (`#/import`, unconditional nav
item â€” unlike the "Your data" backup/restore page, which stays static-only
since it's about browser-local storage specifically). It executes plans
via `web/src/importRunner.ts`, which calls **the same `api` object** every
other page uses (`api.createLot`, `api.updateLot`, `api.transition`) â€”
that client already resolves to either real HTTP calls (server edition)
or direct in-browser `LotService` calls (static edition), so the wizard
needed no edition branching of its own and duplicates no business logic.
`src/lib/import/run.ts` (LotService-based, no HTTP/browser dependency) is
kept for `tests/unit/import.test.ts` and as the natural entry point for a
future server-side bulk-import HTTP endpoint (Â§5).

Every created/updated lot goes through `LotService.create()` /
`LotService.update()` exactly as if a user had filled in the form by
hand, so the usual validation and the new `lot_history` writes (Â§2) apply
automatically â€” an imported lot's Builder/Stage/Owner/Description changes
are just as auditable as a manually-edited one.

### 4.5 XLSX parsing

Uses SheetJS (`xlsx` npm package, added as a real dependency to both the
root and `web` `package.json` â€” not loaded from a CDN, since this is a
bundled Vite app, not an artifact). Only the first sheet is read; cell
values are stringified (`raw: false`) so downstream validation always
sees text, consistent with the CSV path. `xlsx.ts` is dynamically
`import()`-ed from the wizard only when a file is actually detected as
XLSX, so CSV-only users never pull the parser into their bundle.

## 5. Future Procore API integration path

Today, bulk import is a one-shot manual upload â€” there is no live pull
from Procore for the *register* itself (evidence â€” inspections, NCRs,
daily-log quantities â€” already syncs live via webhooks + `SyncService`,
per [architecture.md](architecture.md); only the lot register/spreadsheet
side was manual). A natural v1.4+ extension:

1. **Procore Custom Fields on Checklist Lists / Punch Items as a lot
   source.** If a customer maintains their lot register as a Procore
   custom field (rather than only in the ID/title), a scheduled sync
   could pull those records through the same `ProcoreClient`
   (`src/procore/client.ts`) already used for inspections/NCRs, and feed
   them through the *same* `src/lib/import/plan.ts` â†’ `run.ts` pipeline â€”
   the field-mapping and create-or-update logic doesn't care whether rows
   came from a parsed file or a Procore API page.
2. **A server-side bulk-import endpoint** (`POST
   /api/projects/:id/import`) accepting the same `{mapping, rows}` shape
   the wizard already computes client-side, executing via
   `src/lib/import/run.ts` directly against `PgRepository` instead of
   round-tripping every row through individual HTTP calls from the
   browser â€” meaningful once import volumes grow past what one browser
   session should hold in memory (see Â§7).
3. **Scheduled reconciliation**, mirroring the existing nightly full-pull
   `SyncService.fullSync()` pattern: re-run the same export through the
   wizard's pipeline on a cron, so a customer's external system of record
   and OpenLot's register never drift for more than a sync interval,
   without a human re-uploading a file each time.

None of this requires changing `src/lib/import`'s core contracts
(`FieldMapping`, `ImportRowPlan`, `planImport`, `runImportPlan`) â€” only
where the rows come from and where the plan executes.

| Canonical field | Required | Recognised header aliases (normalised: lowercase, `_`/`-` â†’ space) |

Adding a new importable field is: add it to `ImportField` and
`IMPORT_FIELDS` (`types.ts`), add its aliases to `fields.ts`, read it in
`plan.ts`'s row builder, and forward it in both `run.ts` and
`web/src/importRunner.ts`'s create/update payloads.

## 7. Scalability notes

- **Lot registers are small in absolute terms** â€” civil projects run to
  low thousands of lots (per [architecture.md](architecture.md)'s existing
  scaling note on the register overall). Client-side sort (Â§3) and an
  in-memory import plan (Â§4) are appropriate at this scale; neither would
  be if OpenLot ever targeted programme-of-works scale (tens of thousands
  of lots across many projects).
- **Import is currently bounded by browser memory and one HTTP round
  trip per row** (via `api.createLot`/`updateLot`) in both editions. For
  a few hundred rows this is instant; for a few thousand it's noticeably
  serial (no batching/concurrency in `runImportPlan` â€” deliberately, so
  that one failed row's error message clearly identifies which row, and
  so create-then-transition ordering per row stays simple). If import
  volumes grow, the server-side endpoint in Â§5.2 removes the per-row HTTP
  round trip (same-process calls against `PgRepository` instead), and
  rows could be executed with bounded concurrency.
- **`lot_history` grows monotonically and is never pruned.** At the
  volumes above (thousands of lots, tens of tracked field changes each)
  this is a few hundred thousand rows at most for a large project's
  lifetime â€” trivial for Postgres with the existing
  `(project_id, lot_id, at DESC)` index. If OpenLot ever needed to retain
  history across many years/projects at materially larger scale, the
  natural next step is partitioning `lot_history` by project_id or
  archiving closed projects' history to cold storage, not a redesign of
  the write path.
- **Sorting and the history tab are both purely additive UI reads** â€” no
  N+1 query patterns were introduced: the lot dossier endpoint already
  fetches inspections/ncrs/tests/quantities/claimedIn in parallel
  (`Promise.all`) and `history` was added to the same `Promise.all` (see
  `src/server.ts`), not as a second round trip.
